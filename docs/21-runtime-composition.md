# Runtime Composition Layer
### Sixth Iteration — The substrate that executes railway semantics, trace propagation, and composition wiring

---

## The Missing Substrate

Documents 03, 12, and 18 describe behaviors that are *performed by the composition system*:
- "The composition system injects TraceContext as a side-channel" (doc 18)
- "Every PIPE composition has a declared ErrorHandler" (doc 12)
- "ARUs are unaware of their position in the chain" (doc 12)

But no document specifies what this **composition system** is. Without a substrate, these are aspirations, not architecture. This document defines the runtime composition layer: what it is, what it does, how it is declared, and how it maps to actual code.

---

## What the Runtime Composition Layer Is

The Runtime Composition Layer (RCL) is **not a service or a library in the traditional sense**. It is a **code generation target** — a set of generated wrapper artifacts that implement the composition semantics for each declared composition.

> The RCL turns a composition manifest into executable code that applies railway routing, TraceContext injection, and pattern-specific semantics automatically — without the composed ARUs knowing any of it.

## Mechanism 1: Composition Registry

### What It Is
The Composition Registry is a **build-time artifact** — a generated index of all declared compositions in the codebase, derived from manifests where `composition:` is defined.

### Structure
```typescript
// Generated artifact — do not edit by hand
type CompositionRegistry = {
  [compositionId: ARU_id]: CompositionDefinition
}

type CompositionDefinition = {
  pattern:       CompositionPattern
  params:        PatternParams          // pattern-specific fields from manifest
  errorHandler:  ARU_id | null
  traceEnabled:  boolean                // always true in ARIA
  generated_at:  ISO8601Timestamp
}
```

### How It Is Declared
Compositions are declared in the **manifest** of the orchestrating ARU — the ARU that "is" the composition. There is no separate composition file.

```yaml
# In auth.session.execute.loginFlow manifest:
composition:
  pattern: PIPE
  chain:
    - auth.token.validate.signature
    - auth.token.decode.claims
    - auth.session.create.fromClaims
    - user.profile.load.byId
  error_handler: auth.session.handle.loginError
```

The Generator, when creating a composed ARU, produces both:
1. The manifest with the composition section
2. The **generated composition wrapper** (see §Generated Wrapper Code)

## Mechanism 2: Railway Executor

### What It Does
The Railway Executor is a **generated wrapper function** that:
1. Accepts the chain's input type
2. Calls each ARU in sequence, passing the success output as the next ARU's input
3. On any ARU returning a failure, routes immediately to the error handler
4. Wraps failures in `RailError<E>` with provenance metadata
5. Returns `Result<FinalSuccess, HandledError>` to the caller

### The Generated Wrapper (TypeScript Target)

```typescript
// GENERATED by ARIA build tooling — do not edit
// Composition: auth.session.execute.loginFlow (PIPE)

import { wrapWithProvenance, type RailError } from "@aria/runtime";
import { validateSignature } from "./auth.token.validate.signature";
import { decodeClaims }      from "./auth.token.decode.claims";
import { createFromClaims }  from "./auth.session.create.fromClaims";
import { loadById }          from "./user.profile.load.byId";
import { handleLoginError }  from "./auth.session.handle.loginError";
import type { TokenString, UserSession } from "@domain/auth/types";

export async function loginFlow(
  input: TokenString,
  ctx: TraceContext                          // injected by caller or entry point
): Promise<Result<UserSession, HandledLoginError>> {

  // Railway: each step wraps failure with provenance
  const step1 = await validateSignature(input);
  if (!step1.success) {
    return handleLoginError(wrapWithProvenance(step1.error, "auth.token.validate.signature", ctx));
  }

  const step2 = await decodeClaims(step1.value, ctx);
  if (!step2.success) {
    return handleLoginError(wrapWithProvenance(step2.error, "auth.token.decode.claims", ctx));
  }

  const step3 = await createFromClaims(step2.value, ctx);
  if (!step3.success) {
    return handleLoginError(wrapWithProvenance(step3.error, "auth.session.create.fromClaims", ctx));
  }

  const step4 = await loadById(step3.value, ctx);
  if (!step4.success) {
    return handleLoginError(wrapWithProvenance(step4.error, "user.profile.load.byId", ctx));
  }

  return { success: true, value: step4.value };
}
```

### Key Properties of Generated Code
- **ARUs are pure functions** — they accept typed input, return `Result<T, E>`, know nothing about railway
- **The wrapper is the railway** — all routing logic lives in generated code, never in ARU code
- **Re-generation is safe** — wrapper is always derived from the manifest; editing it by hand is overwritten
- **Language-agnostic logic** — the manifest declares the composition; code generators produce language-specific wrappers

### The `Result<T, E>` Runtime Type

Every ARU returns a `Result` type. This is the two-track type at runtime:

```typescript
// @aria/runtime — core types, never generated
type Result<T, E> =
  | { success: true;  value: T }
  | { success: false; error: E }

type RailError<E> = {
  error:      E
  origin_aru: string           // ARU semantic address
  trace_id:   string           // CorrelationId
  timestamp:  string           // ISO8601
}

function wrapWithProvenance<E>(
  error: E,
  origin: string,
  ctx: TraceContext
): RailError<E> {
  return { error, origin_aru: origin, trace_id: ctx.correlationId, timestamp: new Date().toISOString() }
}
```

### PARTIAL_SUCCESS Track (PARALLEL_JOIN)

For PARALLEL_JOIN with `minimum_required_results`, a third track is added:

```typescript
type ThreeTrack<T, P, E> =
  | { track: 'SUCCESS';        value: T }
  | { track: 'PARTIAL_SUCCESS'; value: P; failures: RailError<E>[] }
  | { track: 'FAILURE';        error: E }
```

Generated wrappers for PARALLEL_JOIN use this three-track type.

## Mechanism 3: Trace Injector

### The Problem
ARUs do not declare `TraceContext` in their contracts — it is a side-channel. But it must be available at every ARU call site to:
- Produce trace records
- Propagate CorrelationId to downstream calls
- Tag RailError with trace_id

### The Solution: Execution Frame
The composition system passes `TraceContext` as an **explicit parameter in generated wrapper code**, not as an ambient global. ARU implementations receive it as a second parameter when called by the wrapper:

```typescript
// ARU signature (declared contract):
function validateSignature(token: TokenString): Result<ValidatedToken, AuthError>

// Wrapper call site (generated by RCL):
const step1 = await validateSignature(input, ctx);
//                                             ^^^
//             ctx is injected by the wrapper — invisible to ARU declaration
```

The ARU *may* use `ctx` for tracing if it calls sub-ARUs. If it doesn't use it, the parameter is ignored. This keeps ARUs independently testable (pass a mock context or null in unit tests).

### TraceContext Creation Rules
```
Entry points (L4/L5 boundary-crossing ARUs) → CREATE new TraceContext
All other ARUs → RECEIVE and PROPAGATE TraceContext from wrapper
```

Generated wrapper code for entry-point ARUs includes:
```typescript
const ctx = createTraceContext({
  originARU: "auth.session.execute.loginFlow",
  correlationId: generateCorrelationId()   // L0 generated value
});
```

## Code Generation Targets

The RCL generates different wrapper code for each supported language target:

| Target | Result Type | Async Model | TraceContext |
|---|---|---|---|
| TypeScript | `Result<T,E>` union | `Promise<Result<T,E>>` | Second parameter |
| Python | `Result[T, E]` (dataclass) | `async def` with `Awaitable` | Keyword argument |
| Rust | `Result<T, E>` (built-in) | `async fn` with `Future` | Second parameter |
| Go | `(T, error)` with wrapped error | standard goroutine | `context.Context` (standard Go pattern) |

All targets implement the same semantics; syntax differs.

## What AI Must Generate vs. What Is Generated

| Artifact | Who Creates It | When |
|---|---|---|
| ARU implementation | GENERATOR agent | During ARU_CREATION subtask |
| ARU manifest | GENERATOR agent (Tier 1/2) + human (Tier 3) | During ARU_CREATION subtask |
| Composition wrapper | **Build tooling** (not AI) | At build time from manifest |
| `@aria/runtime` types | **Framework** (one-time) | Never regenerated |
| TraceContext injection | **Build tooling** | At build time |
| `CompositionRegistry` | **Build tooling** | At build time |

> The Generator writes ARUs and manifests. It does **not** write railway wiring or trace propagation. That is the build tooling's job. This separation keeps Generator context small and ARU implementations pure.

Every step is a build failure if it finds a violation. There are no warnings — only pass or fail.

## Near-Term Implementation Path

The full RCL (automated composition wrapper generation) requires build tooling that does not yet
exist as an ARIA-specific product. The near-term path for teams adopting ARIA today:

### Level 1: Manual Wrappers
Developers write railway wrapper code by hand, following the generated code pattern above.
The manifest still declares the composition. The Reviewer verifies the wrapper matches the manifest.

### Level 2: Template-Based Generation
A simple code generator (e.g., a Node.js/Python script) reads manifests and generates wrappers
from a template per language target. No type system integration — just string templating.

### Level 3: Full RCL
A build-integrated tool with:
- Type system awareness (can verify generated types compile)
- Incremental generation (only re-generates changed compositions)
- Language server plugin (shows composition structure in IDE)
- Test generation integration (see `23-test-infrastructure.md`)

Teams should implement Level 1 immediately, Level 2 within one iteration, and plan for Level 3
as the ARIA adoption matures.
